# Runtime matrix definition

The matrix is the runtime state space rendered per component (PRD §6 Piece 3). Each point
is a **cell** — one concrete rendering configuration. This doc defines the MVP's reduced
matrix and the axes it will grow into.

## MVP: the reduced 8-cell matrix

Axes: **light/dark × default/largest dynamic type × LTR/RTL** = 2 × 2 × 2 = **8 cells**.
Fixed for MVP: iOS simulator, one device size, EN locale.

| # | Appearance | Dynamic type | Direction |
|---|-----------|--------------|-----------|
| 1 | light | default | LTR |
| 2 | light | default | RTL |
| 3 | light | largest | LTR |
| 4 | light | largest | RTL |
| 5 | dark  | default | LTR |
| 6 | dark  | default | RTL |
| 7 | dark  | largest | LTR |
| 8 | dark  | largest | RTL |

**Why these three axes for MVP:** each exposes a class of divergence Figma literally cannot
represent, and together they're cheap to drive on a simulator:
- **light/dark** → token resolution correctness.
- **largest dynamic type** → truncation/overflow the designer never drew ("label truncates
  at largest dynamic type; no frame covers this").
- **LTR/RTL** → mirroring and directional-inset bugs.

## Cell as data (illustrative)

```ts
interface MatrixCell {
  id: string;                    // e.g. "dark.largest.rtl"
  appearance: "light" | "dark";
  dynamicType: "default" | "largest";
  direction: "ltr" | "rtl";
  // fixed-for-MVP context, made explicit so Phase 4 can vary it:
  platform: "ios";
  device: string;                // e.g. "iPhone 15"
  locale: "en";
}

const MVP_MATRIX: MatrixCell[] = /* the 8 rows above, generated combinatorially */;
```

The matrix is generated by taking the cartesian product of the active axes — the same
combinatorial move Piece 4 uses for state enumeration. No model involved.

## How a cell is applied on a simulator (iOS, MVP)

| Axis | Mechanism |
|------|-----------|
| appearance | `xcrun simctl ui <udid> appearance light\|dark` |
| dynamic type | `xcrun simctl ui <udid> content_size <size>` (largest ≈ accessibility XXXL) |
| direction | app-level RTL forcing via `I18nManager` / Expo locale override |

Each cell: apply config → let the screen settle → capture frame out-of-band → store by
hash/handle. (Q-007: iOS-first assumed here; Android mechanisms differ — `adb` + settings.)

## Full-matrix axes (Phase 4, not MVP)

Adds, per PRD §6:
- **OS:** iOS × Android
- **Device size:** smallest × largest
- **Locale:** EN × long-string locale (i18n expansion)

Full cartesian product grows fast (2 OS × 2 appearance × 2 type × 2 size × 2 dir × 2 locale
= 64+). Phase 4 will need cell selection/sampling, not brute force — a note for later, not
an MVP concern.

## Relationship to the diff

Each cell produces one captured frame → one pass-1 diff against the IR intent → optional
escalation. The **manifest aggregates across cells**: `{ cells: 8, mismatches: 2,
crops: [handle, handle] }`. Per-cell pass/fail feeds the eval harness's false-positive
measurement ([phase-0-mvp.md](phase-0-mvp.md)).
